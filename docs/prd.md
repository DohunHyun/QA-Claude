# QA-GPT PRD — AI 기반 IT SI 산출물 자동 품질검토·개선 플랫폼

> 근거 문서: `docs/spec.md`(확정 스펙) · `docs/checklists/`(산출물별 체크리스트 12종) · 실증 세트(품질점검 케이스 가이드 + 시정조치서 PoC 36건).
> 이 PRD는 스펙을 개발 착수용 요구사항 문서로 재구성한 것이며, 상세 판정 기준의 원본은 spec.md·checklists이다.

---

## 네러티브 (Narrative)

「(은행) 투자자문 관리시스템 구축」(코드 `NBIA`)에 3개월차로 투입된 PM 김씨는 분석·설계 산출물을 제출하지만, 결함 피드백은 언제나 며칠 뒤 QA 조직·PMO를 거쳐서야 돌아온다. 문제는 그 결함의 70% 이상이 요구사항 단계 누락처럼 **제출 시점에 곧바로 걸러낼 수 있는 정형 결함**이라는 점이다. 실제로 이 프로젝트의 분석·설계 단계에서만 36건의 시정조치가 나왔고, `Level2명` 오타 하나가 UI목록·메뉴구조도·프로그램목록 3개 산출물로 연쇄 전염되기도 했다. 뒤늦게 발견될수록 수정 비용은 기하급수적으로 커지고, 품질은 PM 개인 역량에 좌우된다.

이제 LLM이 비정형 산출물을 파싱·평가할 뿐 아니라 **결함을 직접 수정한 개선 원본까지** 만들 수 있고, 사내 표준 체크리스트와 PoC 36건이 이미 정답지로 존재한다. QA-GPT의 베팅은 명확하다 — 검토 결과를 '알려주는' 데 그치지 않고, **개선된 산출물 원본 + 시정조치관리대장 + 단계말 검토결과서 3종을 자동 생성**해, AI를 '검토자'가 아니라 PM이 **함께 산출물을 완성하는 파트너**로 만든다. PM이 산출물을 올리면 수 분 안에 무엇이·어디서·왜 문제인지 근거와 함께 돌아오고, 회차를 거듭하며 PM은 품질 기준 자체를 몸에 익힌다. 성공의 모습은 3~4 영업일이 걸리던 단계말 점검이 **수 분으로 압축**되고, PM별 품질 편차가 사라지며, 고객사에는 '검토된 산출물'임을 증명하는 문서가 함께 납품되는 것이다.

## 문제 정의 (Problem Statement)

PM 관점에서:

- 산출물을 제출해도 품질 검토 결과는 **3~4 영업일 뒤**에야 돌아온다(점검 1~2일 + 결과서 발행 3~4일). 실시간 자동 검증 체계가 없다.
- 검토는 제3자 QA·PMO·PM 자체 검토 3가지에만 의존하며, 비용 제약으로 **전수 검토가 불가능**하다.
- 결함의 **70% 이상이 요구사항 단계 누락**에서 기인하나, 제출 시점에 이를 잡아줄 장치가 전무하다.
- 산출물 품질이 **담당 PM의 경험·역량에 과도하게 의존**해 편차가 크고, 신규·중간 투입 PM일수록 결함이 잦다.
- 1개 오류가 여러 산출물로 **연쇄 전염**되어(예: `Level2명` 오타 → UI목록·메뉴구조도·프로그램목록) 뒤늦게 대규모 수정을 유발한다.

## 해결책 (Solution)

PM 관점에서:

- 산출물(HWPX/Excel/PPTX)을 업로드하면 AI가 유형을 자동 인식하고, 해당 단계(관리/분석/설계)·산출물별 체크리스트를 적용해 **수 분 내 전수 검증**한다.
- 검증 결과로 3종 결과물이 자동 생성된다:
  1. **AI 개선 산출물** — 결함을 수정하고 `[개선]` 태그로 변경 위치를 표시한, 원본 포맷(HWPX→HWPX, Excel→Excel)을 유지한 새 버전 (단계별 1개).
  2. **시정조치관리대장** — 개선/권고 항목을 실제 PoC 3시트 스키마(표지·개정이력·시정조치서, 본문 17열)로 정리한 Excel (검증마다 생성).
  3. **단계말 검토결과서** — 항목별 결과(개선/권고/OK)·결함유형·관점·근거를 담은 HWPX 공식 문서 (QA 승인 후 최종).
- 판정은 **개선(ERROR, 필수 수정)** / **권고(WARNING, 권장)** 2단계에 결함유형 6종·점검 관점(산출물/프로세스)을 함께 태깅하며, **모든 판정에 산출물 내 근거 위치(시트명/행/열/ID)를 명시**한다.

## 사용자 스토리 (User Stories)

**PM (프로젝트 관리자)**

1. PM으로서, 검증 전에 프로젝트명·프로젝트코드·단계별 기간을 등록하고 싶다. 그래야 모든 산출물 검증의 1차 기준값으로 쓰인다.
2. PM으로서, HWPX/Excel/PPTX 산출물을 한 번에 여러 개 업로드하고 싶다. 그래야 개별로 올리는 수고를 던다.
3. PM으로서, 업로드한 파일의 산출물 유형이 자동 인식되길 원한다. 그래야 어떤 체크리스트가 적용될지 직접 지정하지 않아도 된다.
4. PM으로서, 검증이 수 분 내 끝나고 진행도가 표시되길 원한다. 그래야 결과를 며칠 기다리지 않는다.
5. PM으로서, 각 결함이 개선인지 권고인지, 결함유형과 근거 위치가 무엇인지 보고 싶다. 그래야 무엇을 먼저 고칠지 판단한다.
6. PM으로서, 결함이 수정된 AI 개선 산출물을 원본 포맷 그대로 받고 싶다. 그래야 양식을 다시 맞추지 않고 검토·수용할 수 있다.
7. PM으로서, `[개선]` 태그로 변경 위치가 표시되길 원한다. 그래야 어디가 바뀌었는지 한눈에 확인한다.
8. PM으로서, 시정조치관리대장으로 대상/완료/잔여 건수와 조치 상태를 추적하고 싶다. 그래야 회차별 수정 현황을 관리한다.
9. PM으로서, 개선 항목이 남아 통과 불가일 때 수정 후 재검증을 무제한 요청하고 싶다. 그래야 통과까지 품질을 끌어올린다.
10. PM으로서, 몇 회차 만에 통과했는지 이력을 보고 싶다. 그래야 개선 추세를 파악한다.
11. PM으로서, 관리산출물이 통과해야 분석단계, 분석이 통과해야 설계단계로 진행되길 원한다. 그래야 결함이 후행 단계로 누적되지 않는다.
12. PM으로서, AI가 근거 없는 빈 항목을 임의로 채우지 않고 결함으로만 보고하길 원한다. 그래야 잘못된 내용이 산출물에 섞이지 않는다.
13. PM으로서, 반복 사용으로 품질 기준을 자연스럽게 학습하고 싶다. 그래야 경험 의존에서 벗어나 역량을 내재화한다.

**QA (품질관리 담당자)**

14. QA로서, 분석/설계 단계 최종 검증 후 단계말 검토결과서를 생성하고 싶다. 그래야 검증 이행 증빙을 남긴다.
15. QA로서, 개선 항목이 남아도 예외 승인으로 통과시킬 수 있길 원한다. 그래야 불가피한 사유를 처리한다.
16. QA로서, 검토결과서를 수기로 확인·서명한 뒤 승인하고 싶다. 그래야 최종 책임 하에 발급된다.
17. QA로서, 검토자가 `AI품질검토봇`으로 기록되고 근거 위치가 명시되길 원한다. 그래야 재현·감리 대응이 가능하다.

**고객사**

18. 고객사로서, 검증 결과 중 수정 사항만 조회하고 싶다. 그래야 필요한 정보만 확인한다.
19. 고객사로서, 검토결과서와 시정조치관리대장을 납품 시 함께 받고 싶다. 그래야 '검토된 산출물'임을 확인하고 검수 부담을 던다.

**관리자**

20. 관리자로서, PM이 신청한 프로젝트를 승인해 활성화하고 싶다. 그래야 검증 절차가 시작된다.
21. 관리자로서, 프로젝트별 단계 진행·회차별 검증 이력·시정조치 현황을 관리하고 싶다. 그래야 전체를 조망한다.

**일반 유저**

22. 일반 유저로서, 프로젝트 현황만 조회하고 싶다. 그래야 진행 상황을 파악한다.

## 구현 결정 (Implementation Decisions)

- **백엔드**: Java 17 + Spring Boot 3.x (사내 표준 스택). REST는 Spring Web + Spring Validation, 대용량 검토는 Spring Async(@Async) + CompletableFuture로 백그라운드 처리해 업로드 응답 지연을 방지한다.
- **프론트**: 정적 HTML/CSS + jQuery (업로드 폼·진행도·다운로드·회차별 현황).
- **LLM 연동**: Claude API를 Spring WebClient 기반 REST로 호출. 유형 인식·4-Phase 검토 프롬프트에 사용.
- **파일 파싱/생성**: Excel·PPTX는 Apache POI, HWPX는 전용 파서/변환. **원본 파일 포맷을 유지**한다(HWPX→HWPX, Excel→Excel).
- **DB/ORM**: PostgreSQL + Spring Data JPA/Hibernate. 체크리스트·검토결과는 **JSONB**로 저장한다.
- **주요 엔티티**: `Project`, `Document`, `ReviewResult`, `Defect`(개선유형·결함유형·관점·근거위치), `CorrectiveAction`(시정조치대장 라인), `ChecklistItem`.
- **서비스 시임(seam)**: 검증 로직은 `service` 하위 5개 모듈로 분리 — `parser`(파싱) → `classifier`(유형 인식) → `checklist`(체크리스트 적용 엔진) → `ai`(Claude 연동) → `generator`(3종 결과물 생성). 검증 정확도의 핵심 시임은 **checklist 엔진**이며, 결과물 품질의 핵심 시임은 **generator**다. 신규 시임은 최소화하고 이 경계에서 테스트한다.
- **검증 실행 순서(4-Phase, 고정)**: Phase 1 문서작성 표준 보증(파일명→표지→개정이력) → Phase 2 요구사항 품질 보증(추적표 기반 누락 체크) → Phase 3 목록/정합성 체크(필수컬럼·ID중복·enum·교차 산출물 정합성) → Phase 4 결과물 생성. 순서를 건너뛰거나 역행하지 않는다.
- **판정 체계**: 심각도 2단계(개선=ERROR / 권고=WARNING), 결함유형 6종(표준미준수·필수항목누락·내용오류·불명확·미제출·중복·기타), 점검 관점 2종(산출물/프로세스)을 태깅. 개선 항목은 통과 불가(QA 예외 승인 시 예외).
- **명명규칙 검증**: 규칙을 코드에 하드코딩하지 않는다. **1순위 = PM 등록 프로젝트정보**(코드·명·기간), **2순위 = 문서작성지침서(`PM-141-01`) 파싱**으로 세부 항목 보충. 프로젝트코드 자릿수·형식은 가변으로 전제.
- **시정조치관리대장 스키마**: 표지·개정이력·시정조치서 3시트. 본문 17열(부적합사항 11 + 시정조치 계획 4 + 시정조치 확인 2). No는 `CA_P##`/`CA_W##`, 검토자는 `AI품질검토봇`. 버전·식별번호 등 표시용 셀은 **텍스트(String)로 강제 지정**해 POI 부동소수점 오차(`1.1000000000000001`)를 방지한다.
- **AI 동작 원칙**: ① 빈 항목은 추측 금지·결함으로만 보고, ② 체크리스트 근거 없는 주관 판단 금지, ③ 파일 구조 임의 변경 금지, ④ 생성 텍스트에 한자 사용 금지(기술 영문 약어는 허용), ⑤ 근거 위치 필수 명시.

## 테스트 결정 (Testing Decisions)

- **좋은 테스트의 기준**: 구현 세부가 아니라 **외부로 관찰 가능한 행위**를 검증한다 — 입력 산출물 → 판정 결과(개선/권고/근거), 입력 산출물 → 생성된 결과물 파일의 내용·구조.
- **회귀 테스트 코퍼스**: **PoC 시정조치서 36건(CA_P01~CA_W34)**을 정답지로 삼아, AI가 동일 결함을 재현·검출하는지로 checklist 엔진의 정확도를 측정한다.
- **테스트 대상 모듈**: `checklist`(판정 정확도), `generator`(3종 결과물이 실제 스키마·포맷을 지키는지 — 특히 시정조치대장 3시트/17열, 텍스트 서식 강제), `classifier`(유형 인식 정확도).
- **E2E**: 업로드 → 검토 → 결과물 다운로드 전 구간을, 11개 산출물 중 3개 이상으로 HWPX/Excel/PPTX 혼합해 검증한다. 단계별 진행 흐름과 원본 포맷 유지를 확인한다.
- **교차 검증**: 목록형↔개별 산출물 간 건수·존재성 정합성(`checklist_artifact_set_consistency.md`)을 별도 케이스로 운용한다.

## 범위 외 (Out of Scope)

MVP(해커톤, ~2026.7.15)에서 제외:

- **파일 형식**: PDF, 이미지 포함 문서 (Phase 2).
- **확장 산출물**: 메뉴구조도(`DS02`)·UI레이아웃(`DS03`)·UI프로그램설계서(`DS08`)·논리/물리 ERD (Phase 2).
- **관리 기능**: PMO 대시보드, PM별 역량 지표 (Phase 2).
- **단계**: 테스트·이행 단계 산출물 검증 (Phase 3, 품질점검 케이스 가이드 4·5장에 기준 존재).
- MVP 검증 대상은 **관리 3 + 분석 4 + 설계 4 = 11개 산출물**로 한정한다.

## 추가 노트 (Further Notes)

- **문서번호 체계**: 관리산출물 `PM-XXX-XX`(지침서 `PM-141-01`, 테일러링 `PM-120-01`, 추적표 `PM-310-01`, 시정조치서 `PM-342-03`), 개발산출물 `NBIA-DV-[번호]`(분석 `AN##`, 설계 `DS##`). `PMO-*` 아님.
