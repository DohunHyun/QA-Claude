# QA-GPT: AI 기반 IT SI 산출물 자동 품질검토 및 개선 플랫폼

**서비스명**: QA-GPT
**핵심 슬로건**: 산출물 업로드 → 품질검토 결과서 + 개선 원본 + 시정조치관리대장 자동 생성
**팀명**: QA-GPT팀
**개발 기간**: 1개월 (해커톤, ~2026.7.15)
**개발 환경**: Java + PostgreSQL + HTML
**실증 기준**: 실제 프로젝트 「2511024\_(은행) 투자자문 관리시스템 구축」(프로젝트코드 `NBIA`)의 QA 품질점검 케이스 가이드 및 시정조치 PoC 데이터에 근거

> **본 문서의 근거 자료 (3종 실증 세트)**
>
> 1. **품질점검 케이스 가이드** (「SI 프로젝트 단계말 QA 품질점검 케이스 가이드」) — 착수·분석·설계·테스트·이행 5단계를 *프로세스 관점*과 *산출물 관점*으로 점검하는 사내 표준 체크리스트 **원문**. → §3 결함 판정 체계, §5 체크리스트 정의의 기준.
> 2. **시정조치서 PoC 산출물** (`시정조치관리대장(분석설계)`) — 위 지침을 실제 산출물에 적용해 생성한 **실제 결함 36건**(CA\_P01~CA\_W34). → §4.4 시정조치관리대장 스키마, §5 결함 예시의 기준.
>
> 세 자료는 **기준(가이드) → 실제 출력(시정조치서)** 로 이어지는 검증된 파이프라인이다. QA-GPT는 이 파이프라인 전체를 자동화·제품화한다.

## 1. 문제 정의 및 배경

### 1.1 IT SI 프로젝트의 산출물 품질관리 구조적 한계

**현황:**

- IT SI 프로젝트는 분석→설계→구현→테스트→안정화에 이르는 다단계 산출물이 연쇄적으로 생산
- 각 단계 산출물 간 정합성 오류가 후행 단계에서 누적되는 구조적 취약성
- **산출물 품질 검토는 현재 3가지 방식에만 의존:**
  - ① 제3자 QA 조직
  - ② PMO 검토
  - ③ PM 자체 검토
- **기존 방식의 문제점:**
  - 검토 시점 지연 (제출 후 며칠 뒤)
  - 전수 검토 불가 (비용 제약)
  - 인력·비용 과다
  - **결함의 70% 이상이 요구사항 단계 누락에서 기인**하나, 제출 시점에 실시간 자동 검증 체계가 전무

**정량 근거 (품질점검 케이스 가이드 기준):**

- 현행 QA 단계말 점검은 **점검 완료  1~2일, 결과서 발행  3~~4일**이 표준 소요 (D-Day 착수 → D+1 완료 → D+2 시정요청 → D+3~4 결과보고)
- 즉, 산출물 1개 단계 점검에 **최소 3~4 영업일**이 소요되며, 이 리드타임 전체가 QA-GPT의 자동화 대상이다.

### 1.2 PM 역량 의존도 심화 및 품질 편차

**현황:**

- 산출물 품질이 담당 PM의 경험·역량에 과도하게 의존
- PM별 품질 편차가 크고 고객사 신뢰도 저하
- 신규/중간 투입 PM의 결함 발생 빈도가 현저히 높음
- 결함 발견 시점이 늦어 수정 비용(Cost of Change) 기하급수적 증가
- 검수·감리 단계에서 뒤늦게 대규모 수정 요구

**실증 근거 (PoC 시정조치서):** 실제 프로젝트 1건의 분석·설계 단계에서만 **36건의 시정조치 항목**이 도출되었으며, 그중 상당수가 명명규칙·필수항목·정합성 등 **AI로 즉시 검증 가능한 정형 결함**이었다. 특히 1개 오류가 여러 산출물로 **연쇄 전염**되는 사례(예: `Level2명` 오타가 UI목록·메뉴구조도·프로그램목록 3개 산출물로 전파)가 확인되어, 조기·전수 자동검증의 가치가 실측되었다.

### 1.3 AI 기술 성숙과 해결 가능성

- LLM(대규모 언어모델) 기술 발전으로 비정형 산출물 문서 파싱·분석·평가·자동 개선 가능
- 단순 검토 결과 알림을 넘어, **결함을 직접 수정한 개선 원본과 시정조치관리대장까지 자동 생성 가능**
- 제3자 QA를 AI로 보강하면서 검토 근거가 문서화되어 감리·검수 대응 자료로 직접 활용 가능
- **사내 표준 체크리스트·양식이 이미 존재**(품질점검 케이스 가이드 + AI 품질점검 지침)하므로, AI가 학습·적용할 품질 기준의 진입장벽이 낮음

### 1.4 해커톤 시현 범위

**포함:** 관리산출물 → 분석단계 산출물 → 설계단계 산출물 검증
**제외:** 테스트, 이행 단계 (향후 개발 — 품질점검 케이스 가이드 4·5장에 기준 존재, Phase 2+ 확장)

## 2. 검증 대상 산출물

> **문서번호 체계 (실제 프로젝트 기준)**
>
> - 관리산출물: `PM-XXX-XX` (예: 문서작성지침서 `PM-141-01`, 테일러링결과서 `PM-120-01`, 요구사항추적표 `PM-310-01`, 시정조치서 `PM-342-03`)
> - 개발산출물: `NBIA-DV-[산출물번호]` — 분석단계 `AN##`, 설계단계 `DS##` (예: 요구사항정의서 `AN06`, 배치Job목록 `AN07`, UI목록 `DS01`, 프로그램목록 `DS07`, 배치설계서 `DS09`, 인터페이스설계서 `DS10`)

### 2.1 관리산출물 (AI 검증 1차 기준)

1. **문서작성지침서** (`PM-141-01`) — PM이 등록한 프로젝트 정보 기반 문서코드 및 명 정의 여부 체크
2. **테일러링결과서** (`PM-120-01`) — 문서작성 지침서 기준 분석/설계 단계별 산출물 명 및 필수 여부 검증
3. **요구사항추적표** (`PM-310-01`) — 분석/설계 단계 산출물의 정합성 검증 대상(양방향 추적성 기준표)

### 2.2 분석단계 산출물 (AI 검증 2차 기준)

1. **요구사항정의서** (`AN06`) — 요구사항추적표와 정합성 검증, 필수사항 체크
2. **프로세스정의서** (`AN04`)— 필수사항 및 구조 체크(흐름 표준, 시작/종료, Input/Output, 액티비티 ID)
3. **인터페이스정의서** (`AN08`)— 필수사항 체크
4. **배치Job목록** (`AN07`) — 필수사항 체크

### 2.3 설계단계 산출물 (AI 검증 3차 기준)

1. **UI목록** (`DS01`) — 요구사항 추적표와 UI ID 검증, Level 단위 구분 체크
2. **프로그램목록** (`DS07`) — UI목록, 인터페이스 정의서, 배치 목록 반영 여부 체크
3. **인터페이스설계서** (`DS10`) — 필수사항 체크, 분석단계 인터페이스정의서 검증
4. **배치설계서** (`DS09`) — 필수사항 체크, 분석단계 배치목록 검증

> **확장 후보 (실제 프로젝트에 존재, MVP 이후 편입):** 메뉴구조도(`DS02`), UI레이아웃(`DS03`, **.pptx**), UI프로그램설계서(`DS08`), 논리/물리 ERD 등. UI레이아웃이 PPT 포맷임에 따라 §4.1 PPT 파싱 지원이 필수다.

### 2.4 점검 관점 — 모든 검증은 2개 관점으로 수행

품질점검 케이스 가이드는 모든 점검 항목을 다음 2개 관점으로 분류한다. QA-GPT의 검증 엔진도 항목별로 관점을 태깅하여 결과서에 반영한다.

| 관점                    | 설명                            | 점검 내용                                             |
| ----------------------- | ------------------------------- | ----------------------------------------------------- |
| **산출물 관점**   | 각 산출물 자체의 품질           | 완전성, 일관성, 정확성, 표준 준수, 필수항목 작성 여부 |
| **프로세스 관점** | 산출물 간 관계 및 관리 프로세스 | 추적성, 변경관리, 승인 절차, 현행화                   |

## 3. 결함 판정 체계

품질점검 케이스 가이드 판정 어휘를 **그대로** 채택한다. (기존 ERROR/WARNING은 UI 배지 별칭으로만 병기)

### 3.1 심각도 = 개선유형 (2단계)

| 심각도(개선유형) | UI 별칭 | 의미                                                                                     | 처리                                  |
| ---------------- | ------- | ---------------------------------------------------------------------------------------- | ------------------------------------- |
| **개선**   | ERROR   | 반드시 시정 또는 시정조치가 필요한 사항 (필수항목 누락, 심각한 불일치, 명명규칙 오류 등) | 통과 불가 — QA 예외 승인 시에만 예외 |
| **권고**   | WARNING | 프로젝트관리자의 판단에 의해 조치 가능한 사항 (일관성 미흡, 상세화 부족 등)              | 보완 권고 (통과 가능)                 |

### 3.2 결함유형 (6종)

| 결함유형                   | 설명                                 | 판정 기준                                            |
| -------------------------- | ------------------------------------ | ---------------------------------------------------- |
| **표준미준수**       | 문서작성 표준·명명규칙 등을 위반    | 표준에서 정의한 규칙과 다른 경우                     |
| **필수항목누락**     | 반드시 포함되어야 하는 항목이 누락   | 표준/가이드가 필수로 지정한 항목이 비어있거나 미작성 |
| **내용오류·불명확** | 기술된 내용이 오류이거나 이해 불명확 | 모순, 오타, 해석 불가능한 내용                       |
| **미제출**           | 산출물 자체가 누락                   | 정의된 산출물 목록에 존재하나 실제 파일 없음         |
| **중복**             | 동일 ID가 복수 존재                  | ID 중복 검증으로 확인                                |
| **기타**             | 개선은 아니나 품질 향상을 위한 권고  | 일관성 미흡, 상세화 부족 등 (권고 유형에 대응)       |

> 명명규칙 위반은 **표준미준수**의 하위 케이스로 판정한다(PoC에서 다수 발생: 날짜 누락, 하이픈 대신 언더스코어, 대/소문자 버전표기 불일치 등).

## 4. 핵심 기능 (자동 검증 및 결과물 생성)

### 4.0 프로젝트 정보 사전 등록 (PM)

**기능:**

- 산출물 검증을 시작하기 전, PM이 프로젝트 기본 정보를 시스템에 직접 등록한다
- 이후 모든 산출물 검증은 본 등록 정보를 **1차 기준값**으로 사용한다 (문서 내용에서 추정·추출한 값보다 우선)

**등록 항목:**

- 프로젝트명 (정식 명칭 — 예: `(은행) 투자자문 관리시스템 구축`)
- 프로젝트코드 (문서번호·파일명에 사용되는 코드 — 자릿수·형식 자유, 예: `NBIA`)
- 프로젝트 기간 (착수일 ~ 종료일) : 분석단계, 설계단계, 구현단계, 테스트단계
- (선택) 단위업무구분 코드표 — 등록 시 문서작성지침서 파싱 결과와 자동 대조

**검증 활용:**

- 모든 산출물 체크리스트의 표지 검증 시 "PM 등록 프로젝트정보 일치 검증"(프로젝트명/프로젝트코드/제·개정일자가 프로젝트 기간 이내인지)을 최우선 **개선(ERROR)** 항목으로 적용
- 문서작성지침서(`PM-141-01`)의 영문 약어명 매핑표(프로젝트코드↔프로젝트명)는 PM 등록정보와 일치하는지 **검증 대상**으로 전환 (기존에는 매핑표 자체가 기준값이었으나, 등록 기능 도입 이후에는 등록정보가 기준값이고 매핑표는 그에 종속)

### 4.1 산출물 파일 업로드 및 자동 파싱

**지원 형식:**

- HWPX (.hwpx) — 한글 문서
- Excel (.xlsx) — 엑셀 파일 (대다수 목록·설계 산출물)
- PPT (.pptx) — UI레이아웃 등 (실제 프로젝트 `DS03` UI레이아웃이 PPT)
- **혼합 다중 업로드 지원** (한 번에 여러 파일 동시 업로드)

**자동 인식 기능:**

- AI가 파일 내용을 자동 분석하여 산출물 유형 자동 인식
- 인식된 산출물 유형 → 검증 단계(관리/분석/설계) 자동 맵핑
- 해당 단계 및 산출물별 체크리스트 자동 적용
- **원본 파일 포맷 유지:** HWPX는 해당 포맷으로, Excel은 Excel로 반환

**명명규칙 검증 기준 — 2단계 우선순위 원칙:**

- 파일명·문서번호 규칙을 코드에 고정값으로 하드코딩하지 않는다
- **1순위**: §4.0에서 PM이 등록한 프로젝트코드·프로젝트명·프로젝트 기간을 검증 기준값으로 사용한다
- **2순위(보조)**: 등록되지 않은 세부 항목(문서관리번호 부여기준, 단위업무구분 코드표, 통제번호 체계 등)은 프로젝트별 **문서작성지침서(`PM-141-01`)를 파싱**하여 보충 추출한다
- 프로젝트코드의 자릿수·형식은 프로젝트마다 다를 수 있음을 전제한다

**실제 명명규칙 패턴 (PoC 기준):**

```
[프로젝트코드]-[도메인]-[산출물번호]-[산출물명]([단위업무])_V[버전]_[YYYYMMDD].[확장자]
예) NBIA-DV-DS09-배치설계서(투자자문)_V1.0a_20260115.xlsx
    NBIA-DV-AN07-배치Job목록(투자자문)_V1.0a_20260115.xlsx
    NBIA-DV-DS10-인터페이스설계서_운용사유형별지표분석(FDTPB203)_v1.0.xlsx
```

PoC에서 실측된 **명명규칙 위반 유형**(모두 표준미준수·개선): ① 날짜(`YYYYMMDD`) 누락, ② 문서번호 분리자로 하이픈(`-`) 대신 언더스코어(`_`) 사용, ③ 버전 표기 소문자 `v1.0` vs 표준 대문자 `V1.0a` 불일치, ④ 동일 문서번호에 비표준 접미사(`_00`) 부가된 중복 파일.

### 4.2 검증 등급 및 기준

§3의 결함 판정 체계를 적용한다.

- **개선(ERROR)** : 즉시 수정 필수 — 필수항목 누락 / 심각한 불일치 / 명명규칙 오류. 통과 불가 (QA 요청으로 예외신청 가능)
- **권고(WARNING)** : 보완 권고 — 품질 기준 미흡 / 일관성 오류 / 명확성 부족. 통과 가능

각 판정에는 **결함유형(6종)**과 **점검 관점(산출물/프로세스)**을 함께 태깅한다.

### 4.3 결과물 ① — AI 개선 산출물 (자동 생성, 단계별 1개)

**기능:**

- 각 검증 단계(관리/분석/설계)별 **AI 개선 산출물 1개씩만 생성**
- 확인된 결함 항목을 수정·보완한 산출물 새 버전 자동 생성
- **원본 파일의 구조·포맷·양식 완벽히 유지**

**출력 형식:**

- **HWPX 원본 형식 유지** (HWPX 입력 → HWPX 출력)
- **Excel 원본 형식 유지** (Excel 입력 → Excel 출력)

**특징:**

- 수정된 부분에 **[개선]** 태그를 삽입하여 변경 위치 명확 표시
- 근거가 없는 빈 항목은 임의로 채우지 않고 §4.6 원칙에 따라 결함으로만 보고 (개선 산출물에는 미기입)
- PM이 개선 원본을 검토·선택적 수용 가능
- **PM의 학습과 역량 내재화 지원**

### 4.4 결과물 ② — 시정조치관리대장 (자동 생성, 매 검증마다)

**기능:**

- 검증마다 자동 생성되는 문서
- 개선/권고 항목의 상태 관리 및 추적

**출력 형식:**

- **Excel (.xlsx) 형식**으로 자동 생성
- **PoC 산출물의 실제 시트/컬럼 구조를 그대로 재현**한다 (아래 스키마 준수)

**시트 구성 (3종 + 참조):**

1. **표지** — 프로젝트명 / 문서명(`시정조치서`) / 단계(`분석/설계`) / 문서번호(`PM-342-03`) / 버전 / 제·개정일자 / 보안 문구
2. **개정이력** — `버전` / `변경일` / `구분`(최초작성·추가·수정·삭제) / `개정내용` / `작성자` / `승인자`
3. **시정조치서** — 상단 요약 + 결함 명세 본문 (아래)

**시정조치서 상단 요약:**

`단계` / `시정조치대상건수` / `시정조치완료건수` / `시정조치잔여건수` / `산출물기준일`

**시정조치서 본문 컬럼 (3개 그룹, 17열):**

| 그룹                    | 컬럼                                                                                                               |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------ |
| **부적합사항**    | No · 업무명 · 구분 · 검토일 · 검토자 · 산출물명 · 파일명 · 부적합 위치 · 개선 유형 · 결함유형 · 결함내용 |
| **시정조치 계획** | 조치담당자 · 완료예정일 · 조치완료일 · 비고(시정조치계획)                                                       |
| **시정조치 확인** | 조치확인자 · 조치확인일                                                                                           |

- **No 형식**: `CA_P##`(공통·프로젝트관리) / `CA_W##`(개발산출물). 검토자는 `AI품질검토봇`으로 기록.
- **부적합 위치**는 산출물 내 근거(시트명/Row/열/ID)를 반드시 명시 (예: `배치Job목록(시트: 내용, Row 3~62)`).

**구현 주의사항:**

- 버전·식별번호 등 표시용 컬럼(예: `1.1`)은 셀 서식을 **텍스트(String)로 강제 지정**하여 저장한다. 숫자 서식으로 저장 시 Apache POI/Excel이 `1.1000000000000001`과 같은 부동소수점 오차를 표시하는 문제가 실제 PoC에서 확인됨.

**활용 용도:**

- 회차별 검증 이력 관리 / PM 수정 현황 추적 / 최종 QA 검증 시 증빙 자료

### 4.5 결과물 ③ — 단계말 검토결과서 (QA 승인 후, 최종)

**기능:**

- 분석/설계 단계 최종 검증 완료 후 QA가 생성
- 검증 이행 증빙 자료
- 모든 검증 단계의 항목별 검토 결과 및 결함 내용 포함

**출력 형식:**

- **HWPX (.hwpx) 형식**으로 작성 — 사내 표준 한글 문서 양식 사용

**포함 내용:**

- 검증 대상 산출물 목록
- 단계별 검증 결과 요약
- 항목별 검토 결과 (개선/권고/OK) 및 결함유형
  - 결함유형 분류 (표준미준수 / 필수항목누락 / 내용오류·불명확 / 미제출 / 중복 / 기타)
  - 점검 관점 (산출물 / 프로세스)
  - 구체적 결함 내용 및 근거 (산출물 내 위치: 시트명/행/열/ID 등)
  - 개선 권고사항 (실행 가능한 구체적 수정 방향)
- 최종 검증 평가 / QA 검증자 서명

**활용 용도:** 고객사 제출(검증 이행 증빙) / 계약 이행 근거 / 감리·검수 대응 자료

### 4.6 AI 동작 원칙 및 금지사항

**금지 사항:**

1. 산출물 내용이 없는 항목에 대해 임의의 내용을 채우거나 추측하지 않는다 — 빈 항목은 추측 대신 **결함으로 보고**한다
2. 표준이나 가이드라인(체크리스트)에 근거하지 않은 주관적 판단을 하지 않는다 — 검증 가능한 근거 기반 판단만 적용
3. 파일 구조를 임의로 변경하지 않는다 — AI 개선 산출물 생성 시에도 기존 구조를 유지하고 결함 수정만 수행

**판정 4대 기본 원칙**

1. **증거 기반 판단** — 모든 결함 식별은 실제 산출물 내용과의 비교로 입증 가능해야 한다
2. **객관성** — 표준·가이드라인에 근거하여 판단한다
3. **완전성** — 점검 리스트의 모든 항목을 누락 없이 확인한다
4. **재현 가능성** — 다른 QA 담당자가 동일 기준으로 동일 결과를 도출할 수 있어야 한다

**주의 사항:**

1. AI가 생성하는 모든 텍스트에서 한자(漢字) 사용을 금지한다
2. 기술 용어는 영문 허용 (PK, FK, ERD, API, UI, UX 등)
3. 판정 시 산출물 내 근거 위치(시트명/행/열/ID 등)를 반드시 명시한다

이 원칙은 §4.3(AI 개선 산출물), §4.4(시정조치관리대장), §4.5(단계말 검토결과서) 생성 로직 전체에 공통 적용된다.

## 7. 업무 프로세스 (PM 기준)

### 7.1 전체 흐름

```
1. 프로젝트 신청 (PM)
   → 프로젝트명, 코드명(예: NBIA), 단계별 일자, 기타 정보

2. 관리자 승인

3. 관리산출물 등록 및 검증
   ├─ 문서작성지침서(PM-141-01) 업로드

4. 관리산출물 검증 완료 후

5. 분석단계 산출물 등록 및 검증 (개별 검증 가능)
   ├─ 요구사항정의서 / 프로세스정의서 / 인터페이스정의서 / 배치Job목록 업로드
   └─ AI 검증 → AI 개선 산출물(1개) + 시정조치관리대장 생성

6. 분석단계 전체 검증 완료 후
   → 설계단계 진행 가능 (최종 버전 산출물 기반)

7. 설계단계 산출물 등록 및 검증 (개별 검증 가능)
   ├─ UI목록 / 프로그램목록 / 인터페이스설계서 / 배치설계서 업로드
   └─ AI 검증 → AI 개선 산출물(1개) + 시정조치관리대장 생성

8. 설계단계 전체 검증 완료 후
   → 단계말 검토결과 요청 가능
   → QA 수기 확인 → 승인 시 검토 결과서 발급

9. 최종 검토 결과서 발급
   → 고객사 제출 (검증 이행 증빙)
```

### 7.2 산출물별 검증 시 고정 실행 순서 (4-Phase)

각 산출물(관리/분석/설계 불문)을 검증할 때 AI는 다음 4단계 순서를 반드시 준수한다. 순서를 건너뛰거나 역행하지 않는다. (Agents 지침 §6 실행계획과 동일)

1. **Phase 1. 문서작성 표준 보증** — 파일명 검증 → 표지 검증 → 개정이력 검증
2. **Phase 2. 요구사항 품질 보증** — 요구사항추적표 기반 누락 체크 (요구사항 연계 산출물)
3. **Phase 3. 단순 목록/정합성 체크** — 필수컬럼·ID중복·enum값·교차 산출물 정합성(배치/인터페이스/UI/프로그램) 검증
4. **Phase 4. 결과 산출물 생성** — 단계말 검토결과서 + AI 개선 산출물 + 시정조치관리대장 생성

**각 회차별 검증:**

- 개선(ERROR) 항목이 있으면 통과 불가 → PM이 수정 후 재검증 요청 → 반복 검증으로 품질 개선 추적

**회차별 산출물:**

- AI 개선 산출물 (원본 포맷 유지, 단계별 1개만 생성)
- 시정조치관리대장 (Excel, 검증마다 생성)
- 단계말 검토결과서 (HWPX, 최종 승인 후 생성 — 모든 검증 결과 포함)

> **자동화 목표**: 현행 QA 단계말 점검의 표준 리드타임(점검 1~2일 + 결과서 3~4일)을 **수 분 이내**로 압축한다.

## 9. 기대 효과 및 차별성

### 9.1 직접적 업무 효과

| 효과                       | 현행                       | QA-GPT             |
| -------------------------- | -------------------------- | ------------------ |
| **검토 소요 시간**   | 점검 1~2일 + 결과서 3~4일 | 수 분              |
| **24시간 검토**      | ✗                         | ✓                 |
| **검토 비용**        | 제3자 QA 외부 비용         | 자동화로 대폭 감소 |

| **개선 산출물**      | 수동 작성                  | 자동 생성          |
| **고객사 검수 대응** | 분쟁 위험                  | 문서화된 근거 제출 |

**정량적 기대 효과:**

- 검토 시간: 기존 3~4 영업일 → **수 분 (90%+ 단축)**

- 결함 조기 발견 비용: **Cost of Change 기하급수적 감소** (PoC에서 1개 오류의 3중 연쇄 전염 실측)

- AI가 동일한 품질 기준으로 전수 검토 / PM별 산출물 품질 편차 제거 / 일관된 납품 품질 보장

### 9.3 고객사 신뢰도 향상

- 품질검토 결과서 + 시정조치관리대장을 납품 시 함께 제출 → '검토된 산출물'임을 증명
- 고객사의 검수 부담 경감 / 당사 브랜드 신뢰성 향상

## 10. 기술 스택 및 구현 방법

### 10.1 기술 선택 근거

| 계층                  | 기술                                      | 선택 사유                                                                      |
| --------------------- | ----------------------------------------- | ------------------------------------------------------------------------------ |
| **Backend**     | Java 17 + Spring Boot 3.x                 | 사내 표준 스택 정합성, 안정성·유지보수성, 엔터프라이즈 적합                   |
| **API**         | Spring Web (REST) + Spring Validation     | 표준 REST API 설계, 입력 검증 일원화                                           |
| **비동기 처리** | Spring Async (@Async) + CompletableFuture | 대용량 산출물 검토 백그라운드 처리, 업로드 응답 지연 방지                      |
| **Frontend**    | 정적 HTML/CSS + jQuery                    | 간단한 웹 UI (업로드 폼, 진행도, 다운로드 링크)                                |
| **LLM 연동**    | Claude API (Spring WebClient 기반 REST)   | 문서 분석·텍스트 생성 최고 수준                                               |
| **파일 파싱**   | Apache POI (Excel/PPT) + HWPX 파서        | Excel·PPTX: Apache POI, HWPX: 전용 파서/변환                                  |
| **ORM/DB 연동** | Spring Data JPA + Hibernate               | 검토 이력·결함 항목 엔티티 매핑 표준화                                        |
| **Database**    | PostgreSQL                                | JSONB 기반 체크리스트/검토결과 저장 유리, 라이선스 비용 없음, SaaS화 확장 용이 |
| **빌드 도구**   | Gradle (Kotlin DSL)                       | 빠른 빌드, 멀티모듈 구성 용이                                                  |
| **API 문서화**  | springdoc-openapi (Swagger UI)            | 프론트 협업 및 API 명세 자동화                                                 |

### 10.2 구현 아키텍처 (개념도)

```
사용자 (PM/QA/관리자/고객사)
        ↓
    [웹 UI - 정적 HTML/CSS + jQuery]
  · 파일 업로드 폼 · 검토 진행도 · 결과물 다운로드 · 회차별 현황
        ↓ REST API (JSON / Multipart)
    [Backend API - Spring Boot]
  · Controller: 업로드/조회/다운로드 엔드포인트
  · Service: 파일 관리, 검토 오케스트레이션
  · File Parser (Apache POI/HWPX): HWPX/Excel/PPTX 파싱
        ↓
    [AI 검토 엔진 - Service Layer]
  · 산출물 유형 인식 (Claude API 호출)
  · 검증 단계 맵핑 (관리/분석/설계)
  · 체크리스트 기반 4-Phase 검증 (§7.2)
  · 개선/권고 등급 + 결함유형(6종) + 관점 태깅
        ↓
    [결과물 생성 - Document Generator]
  1. AI 개선 산출물 (단계별 1개, 원본 포맷 유지: HWPX/Excel)
  2. 시정조치관리대장 (매 검증마다, Excel — 표지/개정이력/시정조치서 3시트)
  3. 단계말 검토결과서 (최종, HWPX — 모든 검증 결과 포함)
        ↓
    [Database - PostgreSQL / Spring Data JPA]
  · 검토 이력 · 결함 항목 · 회차별 현황 · 시정조치 상태 추적
        ↓
    [현황 조회]
  · 프로젝트별 단계 진행 · 회차별 검증 이력 · 시정조치 현황(대상/완료/잔여)
```

- 공공·금융·제조 등 발주 유형별 특화 체크리스트
- 검토 이력 분석 (반복 오류 패턴 학습, 조직 역량 추이)

### 13.3 장기 확장 (Phase 4+)

- **SaaS 서비스화** (타 SI 기업 제공)
- **PM 역량 컨설팅 상품** (QA-GPT 기반 교육)
- **산업별 특화 패키지** (공공·금융·제조·통신·보험)
- **API 개방 및 고객사 자체 시스템 연계**

## 14. 평가 기준별 부합성

### 14.1 혁신성 ⭐⭐⭐⭐⭐

- 단순 검토 자동화 → **3종 결과물(AI 개선산출물 + 시정조치대장 + 단계말검토결과서) 자동 생성**
- 관리·분석·설계 단계별 **체계적 검증 프로세스 자동화**
- PM이 AI를 '검토자'가 아닌 **'함께 산출물을 완성하는 파트너'로 활용**
- 국내 SI 업계에서 **전례 없는 접근법**

### 14.2 실현 가능성 ⭐⭐⭐⭐⭐

- Claude API + Apache POI 등 **검증된 기술 조합으로 즉시 구현 가능**
- **사내 표준 체크리스트·양식이 이미 존재**(품질점검 케이스 가이드 + AI 품질점검 지침) → 품질 기준 진입장벽 낮음
- **실제 PoC(시정조치서 36건)로 파이프라인이 이미 실증됨** → 재현·자동화가 핵심 과제
- Java/Spring Boot는 사내 표준 스택, PostgreSQL은 오픈소스로 라이선스 비용 없이 JSONB 기반 저장에 적합

### 14.3 업무 적용성 ⭐⭐⭐⭐⭐

- 현재 진행 중인 **모든 IT SI 프로젝트에 즉시 파일럿 적용 가능**
- PM·QA·관리자 모두 **실무에서 매일 사용하는 도구** / 학습 비용 없음 (기존 파일 업로드 방식)
- 품질검토 프로세스에 **즉시 통합 가능**

### 14.4 확장 가능성 ⭐⭐⭐⭐

- **단기**: 형식 확장, 확장 산출물, PMO 대시보드
- **중기**: 테스트/이행 단계, 발주 유형별 특화
- **장기**: SaaS 서비스화, PM 컨설팅 상품, 산업별 패키지
