# QA-GPT: AI 기반 IT SI 산출물 자동 품질검토 및 개선 플랫폼

**서비스명**: QA-GPT
**핵심 슬로건**: 산출물 업로드 → 품질검토 결과서 + 개선 원본 + 시정조치관리대장 자동 생성
**팀명**: QA-GPT팀
**개발 기간**: 1개월 (해커톤, ~2026.7.15)
**개발 환경**: Java + PostgreSQL + HTML

---

## 1. 문제 정의 및 배경

### 1.1 IT SI 프로젝트의 산출물 품질관리 구조적 한계

**현황:**

- IT SI 프로젝트는 분석→설계→구현→테스트→안정화에 이르는 다단계 산출물이 연쇄적으로 생산
- 각 단계 산출물 간 정합성 오류가 후행 단계에서 누적되는 구조적 취약성
- **산출물 품질 검토는 현재 3가지 방식에만 의존:**
  - ① 제3자 QA 조직
  - ② PMO 검토
  - ③ PM 자체 검토
- **기존 방식의 문제점:**
  - 검토 시점 지연 (제출 후 며칠 뒤)
  - 전수 검토 불가 (비용 제약)
  - 인력·비용 과다
  - **결함의 70% 이상이 요구사항 단계 누락에서 기인**하나, 제출 시점에 실시간 자동 검증 체계가 전무

### 1.2 PM 역량 의존도 심화 및 품질 편차

**현황:**

- 산출물 품질이 담당 PM의 경험·역량에 과도하게 의존
- PM별 품질 편차가 크고 고객사 신뢰도 저하
- 신규/중간 투입 PM의 결함 발생 빈도가 현저히 높음
- 결함 발견 시점이 늦어 수정 비용(Cost of Change) 기하급수적 증가
- 검수·감리 단계에서 뒤늦게 대규모 수정 요구

### 1.3 AI 기술 성숙과 해결 가능성

- LLM(대규모 언어모델) 기술 발전으로 비정형 산출물 문서 파싱·분석·평가·자동 개선 가능
- 단순 검토 결과 알림을 넘어, **결함을 직접 수정한 개선 원본과 시정조치관리대장까지 자동 생성 가능**
- 제3자 QA를 AI로 보강하면서 검토 근거가 문서화되어 감리·검수 대응 자료로 직접 활용 가능

### 1.4 해커톤 시현 범위

**포함:** 관리산출물 → 분석단계 산출물 → 설계단계 산출물 검증  
**제외:** 테스트, 이행 단계 (향후 개발)

## 2. 검증 대상 산출물

### 2.1 관리산출물 (AI 검증 1차 기준)

1. **문서작성지침서** - PM이 등록한 프로젝트 정보 기반 문서코드 및 명 정의 여부 체크
2. **테일러링결과서** - 문서작성 지침서 기준 분석/설계 단계별 산출물 명 및 필수 여부 검증
3. **요구사항추적표** - 분석/설계 단계 산출물의 정합성 검증 대상

### 2.2 분석단계 산출물 (AI 검증 2차 기준)

1. **요구사항정의서** - 요구사항추적표와 정합성 검증, 필수사항 체크
2. **프로세스정의서** - 필수사항 및 구조 체크
3. **인터페이스정의서** - 필수사항 체크
4. **배치Job목록** - 필수사항 체크

### 2.3 설계단계 산출물 (AI 검증 3차 기준)

1. **UI목록** - 요구사항 추적표와 UI ID 검증, Level 단위 구분 체크
2. **프로그램목록** - UI목록, 인터페이스 정의서, 배치 목록 반영 여부 체크
3. **인터페이스설계서** - 필수사항 체크, 분석단계 인터페이스정의서 검증
4. **배치설계서** - 필수사항 체크, 분석단계 배치목록 검증

## 3. 핵심 기능 (자동 검증 및 결과물 생성)

### 3.0 프로젝트 정보 사전 등록 (PM)

**기능:**

- 산출물 검증을 시작하기 전, PM이 프로젝트 기본 정보를 시스템에 직접 등록한다
- 이후 모든 산출물 검증은 본 등록 정보를 **1차 기준값**으로 사용한다 (문서 내용에서 추정·추출한 값보다 우선)

**등록 항목:**

- 프로젝트명 (정식 명칭)
- 프로젝트코드 (문서번호·파일명에 사용되는 코드 — 자릿수·형식 자유, 예: 3자리/4자리 등)
- 프로젝트 기간 (착수일 ~ 종료일)
- (선택) 단위업무구분 코드표 — 등록 시 문서작성지침서 파싱 결과와 자동 대조

**검증 활용:**

- 모든 산출물 체크리스트의 표지 검증 시 "PM 등록 프로젝트정보 일치 검증"(프로젝트명/프로젝트코드/제·개정일자가 프로젝트 기간 이내인지)을 최우선 ERROR 항목으로 적용
- 문서작성지침서(PM-141-01)의 영문 약어명 매핑표(프로젝트코드↔프로젝트명)는 PM 등록정보와 일치하는지 **검증 대상**으로 전환 (기존에는 매핑표 자체가 기준값이었으나, 등록 기능 도입 이후에는 등록정보가 기준값이고 매핑표는 그에 종속)

---

### 3.1 산출물 파일 업로드 및 자동 파싱

**지원 형식:**

- HWP/HWPX (.hwp, .hwpx) - 한글 문서
- Excel (.xlsx) - 엑셀 파일
- **혼합 다중 업로드 지원** (한 번에 여러 파일 동시 업로드)

**자동 인식 기능:**

- AI가 파일 내용을 자동 분석하여 산출물 유형 자동 인식
- 인식된 산출물 유형 → 검증 단계(관리/분석/설계) 자동 맵핑
- 해당 단계 및 산출물별 체크리스트 자동 적용
- **원본 파일 포맷 유지:** HWP/HWPX는 해당 포맷으로, Excel은 Excel로 반환

**명명규칙 검증 기준 — 2단계 우선순위 원칙:**

- 파일명·문서번호 규칙(`NH[프로젝트코드]-...` 등)을 코드에 고정값으로 하드코딩하지 않는다
- **1순위**: §3.0에서 PM이 등록한 프로젝트코드·프로젝트명·프로젝트 기간을 검증 기준값으로 사용한다
- **2순위(보조)**: 등록되지 않은 세부 항목(문서관리번호 부여기준, 단위업무구분 코드표, 통제번호 체계 등)은 프로젝트별 **문서작성지침서(PM-141-01)를 파싱**하여 보충 추출한다
- 프로젝트코드의 자릿수·형식은 프로젝트마다 다를 수 있음을 전제한다 (예: 3자리 코드 외에 4자리 코드 등 변형 존재 가능)

---


### 3.2 검증 등급 및 기준

**2가지 등급 부여 (Error/Warning):**

- **ERROR** : 즉시 수정 필수
  - 필수항목 누락
  - 심각한 불일치
  - 명명규칙 오류
  - 통과 불가능 (QA 요청으로 예외신청 가능)

- **WARNING** : 보완 권고
  - 품질 기준 미흡
  - 일관성 오류
  - 명확성 부족

### 3.3 결과물 ① — 품질검토 결과서 (자동 생성)

**기능:**

- AI가 사내 표준 체크리스트 대비 항목별 검토 수행
- 각 항목에 대해 ERROR/WARNING/OK 3단계 등급 부여
- **결함유형 분류 적용** (Agents.md §1.6 기준): 표준미준수 / 필수항목누락 / 내용오류·불명확 / 미제출 / 중복 / 기타(권고)
  - ERROR 등급 항목은 위 6종 결함유형 중 하나로 반드시 분류
  - "기타" 유형은 권고(WARNING) 등급에만 적용

**출력 형식:**

- **Excel (.xlsx) 형식**으로 자동 생성
- 항목별 포함 내용 (Agents.md §7.1 결함 기록 양식 기준):
  - 결함 번호 (자동 부여, 예: DEF-001)
  - 산출물명
  - 점검 항목 (체크리스트 상의 검증 항목명)
  - 검토 결과 (ERROR/WARNING/OK)
  - 결함유형 (표준미준수/필수항목누락/내용오류·불명확/미제출/중복/기타)
  - 결함 내용 (구체적 설명 — 예: "요구사항 ID REQ-003 중복" 수준의 구체성, "ID 오류 있음" 같은 모호한 표현 금지)
  - 근거 (산출물 내 위치: 시트명/행/열/ID 등)
  - 개선 권고사항 (실행 가능한 구체적 수정 방향)

**품질 점수:**

- AI가 자동으로 **0~100점 산출**
- 프로젝트 품질 수준을 정량적으로 가시화

---

### 3.4 결과물 ② — AI 개선 산출물 (자동 생성, 단계별 1개)

**기능:**

- 각 검증 단계(관리/분석/설계)별 **AI 개선 산출물 1개씩만 생성**
- 확인된 결함 항목을 수정·보완한 산출물 새 버전 자동 생성
- **원본 파일의 구조·포맷·양식 완벽히 유지**

**출력 형식:**

- **HWP/HWPX 원본 형식 유지** (HWP 입력 → HWPX 출력)
- **Excel 원본 형식 유지** (Excel 입력 → Excel 출력)

**특징:**

- 수정된 부분에 **[개선]** 태그를 삽입하여 변경 위치 명확 표시
- PM이 개선 원본을 검토·선택적 수용 가능
- **PM의 학습과 역량 내재화 지원**

---

### 3.5 결과물 ③ — 시정조치관리대장 (자동 생성, 매 검증마다)

**기능:**

- 검증마다 자동 생성되는 문서
- ERROR/WARNING 항목의 상태 관리 및 추적

**출력 형식:**

- **Excel (.xlsx) 형식**으로 자동 생성
- **구현 주의사항**: 버전·식별번호 등 표시용 컬럼(예: "1.1")은 셀 서식을 **텍스트(String)로 강제 지정**하여 저장한다. 숫자 서식으로 저장 시 Apache POI/Excel이 `1.1000000000000001`과 같은 부동소수점 오차를 표시하는 문제가 실제 PoC에서 확인됨

**포함 내용:**

- 검증 회차 / 검증 날짜
- 산출물명
- 발견 항목 (ERROR/WARNING)
- 조치 상태 (미완료/진행중/완료)
- 조치 내용
- 담당자 / 완료 예정일

**활용 용도:**

- 회차별 검증 이력 관리
- PM의 수정 현황 추적
- 최종 QA 검증 시 증빙 자료

### 3.6 AI 동작 원칙 및 금지사항 (Agents.md §8 기준)

**금지 사항:**

1. 산출물 내용이 없는 항목에 대해 임의의 내용을 채우거나 추측하지 않는다 — 빈 항목은 추측 대신 **결함으로 보고**한다
2. 표준이나 가이드라인(체크리스트)에 근거하지 않은 주관적 판단을 하지 않는다 — 검증 가능한 근거 기반 판단만 적용
3. 파일 구조를 임의로 변경하지 않는다 — AI 개선 산출물 생성 시에도 기존 구조를 유지하고 결함 수정만 수행

**주의 사항:**

1. AI가 생성하는 모든 텍스트에서 한자(漢字) 사용을 금지한다
2. 기술 용어는 영문 허용 (PK, FK, ERD, API, UI, UX 등)
3. 판정 시 산출물 내 근거 위치(시트명/행/열/ID 등)를 반드시 명시한다

이 원칙은 §3.3(품질검토 결과서), §3.4(AI 개선 산출물) 생성 로직 전체에 공통 적용된다.

---

### 3.7 최종 결과물 — 단계말 검토결과서 (QA 승인 후)

**기능:**

- 분석/설계 단계 최종 검증 완료 후 QA가 생성
- 검증 이행 증빙 자료

**출력 형식:**

- **HWPX (.hwpx) 형식**으로 작성
- 사내 표준 한글 문서 양식 사용

**포함 내용:**

- 검증 대상 산출물 목록
- 단계별 검증 결과 요약
- 최종 품질 평가
- QA 검증자 서명

**활용 용도:**

- 고객사 제출 (검증 이행 증빙)
- 계약 이행 근거
- 감리/검수 대응 자료

## 4. 시스템 사용자 및 역할

### 4.1 5가지 사용자 유형

1. **PM (프로젝트 관리자)** - 산출물 등록 및 검증 요청
2. **고객사** - 검증 결과 확인 (수정 사항만 조회 가능)
3. **QA (품질관리 담당자)** - 최종 검증 및 예외 승인
4. **관리자** - 프로젝트 승인 및 단계별 현황 관리
5. **일반 유저** - 프로젝트 현황만 조회 가능

## 5. 업무 프로세스 (PM 기준)

### 5.1 전체 흐름

```
1. 프로젝트 신청 (PM)
   → 프로젝트명, 코드명, 단계별 일자, 기타 정보

2. 관리자 승인
   → 프로젝트 활성화

3. 관리산출물 등록 및 검증
   ├─ 문서작성지침서 업로드
   ├─ 테일러링 결과서 업로드
   ├─ 요구사항 추적표 업로드
   └─ AI 검증 → 품질검토 결과서 + AI 개선 산출물 + 시정조치관리대장 생성
   
   ⚠ ERROR 존재 시 QA 예외신청 가능

4. 관리산출물 검증 완료 후
   → 분석단계 진행 가능
   → 최종 버전 산출물 기반으로 분석단계 실시

5. 분석단계 산출물 등록 및 검증 (개별 검증 가능)
   ├─ 요구사항정의서 업로드
   ├─ 프로세스정의서 업로드
   ├─ 인터페이스정의서 업로드
   ├─ 배치Job목록 업로드
   └─ AI 검증 (각 산출물별) → 품질검토 결과서 + AI 개선 산출물 (1개) + 시정조치관리대장 생성

6. 분석단계 전체 산출물 검증 완료 후
   → 설계단계 진행 가능
   → 최종 버전 산출물 기반으로 설계단계 실시

7. 설계단계 산출물 등록 및 검증 (개별 검증 가능)
   ├─ UI목록 업로드
   ├─ 프로그램목록 업로드
   ├─ 인터페이스설계서 업로드
   ├─ 배치설계서 업로드
   └─ AI 검증 (각 산출물별) → 품질검토 결과서 + AI 개선 산출물 (1개) + 시정조치관리대장 생성

8. 설계단계 전체 산출물 검증 완료 후
   → 단계말 검토결과 요청 가능
   → QA 수기 확인 → 승인 시 검토 결과서 발급

9. 최종 검토 결과서 발급
   → 고객사 제출 (검증 이행 증빙)
```

### 5.2 검증 프로세스

**산출물별 검증 시 고정 실행 순서 (Agents.md §3, §8.3 기준):**

각 산출물(관리/분석/설계 불문)을 검증할 때 AI는 다음 4단계 순서를 반드시 준수한다. 순서를 건너뛰거나 역행하지 않는다.

1. **Phase 1. 문서작성 표준 보증** — 파일명 검증 → 표지 검증 → 개정이력 검증
2. **Phase 2. 요구사항 품질 보증** — 요구사항추적표 기반 누락 체크 (해당 산출물이 요구사항과 연계되는 경우)
3. **Phase 3. 단순 목록/정합성 체크** — 체크리스트 본문의 필수컬럼·ID중복·enum값·교차 산출물 정합성 검증
4. **Phase 4. 결과 산출물 생성** — 품질검토 결과서 + AI 개선 산출물 + 시정조치관리대장 생성

**각 회차별 검증:**

- ERROR 항목이 있으면 통과 불가
- PM이 수정 후 재검증 요청
- 반복 검증으로 품질 개선 추적

**회차별 산출물:**

- 품질검토 결과서 (Excel, 검증마다 생성)
- AI 개선 산출물 (원본 포맷 유지, 단계별 1개만 생성)
- 시정조치관리대장 (Excel, 검증마다 생성)
- 단계말 검토결과서 (HWPX, 최종 승인 후 생성)

## 8. 기술 스택 및 구현 방법

### 8.1 기술 선택 근거

| 계층                        | 기술                                         | 선택 사유                                                              |
| --------------------------- | -------------------------------------------- | ---------------------------------------------------------------------- |
| **Backend**           | Java 17 + Spring Boot 3.x                    | 사내 표준 스택과의 정합성, 안정성과 유지보수성, 엔터프라이즈 환경 적합 |
| **API**               | Spring Web (REST) + Spring Validation        | 표준화된 REST API 설계, 입력 검증 일원화                               |
| **비동기 처리**       | Spring Async (@Async) + CompletableFuture    | 대용량 산출물 검토를 백그라운드로 처리, 업로드 응답 지연 방지          |
| **Frontend**          | 정적 HTML/CSS + jQuery                       | 간단한 웹 UI (파일 업로드 폼, 검토 진행도, 결과물 다운로드 링크)       |
| **LLM 연동**          | Claude API (Spring WebClient 기반 REST 연동) | 문서 분석, 텍스트 생성 최고 수준                                       |
| **파일 파싱**         | Apache POI (Excel) + XWPF (HWPX)            | Excel: Apache POI, HWPX: Apache POI-HWPF 또는 문서 변환 라이브러리   |
| **ORM/Database 연동** | Spring Data JPA + Hibernate                  | 검토 이력·품질 점수 등 엔티티 매핑 표준화                             |
| **Database**          | PostgreSQL                                   | JSONB 기반 체크리스트/검토결과 저장에 유리, 오픈소스 라이선스 비용 없음, 향후 SaaS화 시 확장 용이                    |
| **빌드 도구**         | Gradle (Kotlin DSL)                          | 빠른 빌드, 멀티모듈 구성 용이                                          |
| **API 문서화**        | springdoc-openapi (Swagger UI)               | 프론트엔드 협업 및 API 명세 자동화                                     |

        ↓

## 10. 성공 기준 (2026.7.15 해커톤 시연)

해커톤 종료 시점에서 다음이 **실제 동작**으로 시연 가능해야 함:

### 🎯 **필수 성공 기준 (MVP)**

1. ✅ **파일 업로드 및 AI 검토** (HWP/HWPX/Excel 지원)

   - **관리**: 문서작성지침서, 테일러링 결과서, 요구사항 추적표
   - **분석**: 요구사항정의서, 프로세스정의서, 인터페이스정의서, 배치Job목록
   - **설계**: UI목록, 프로그램목록, 인터페이스설계서, 배치설계서
   - AI가 5분 이내 검토 완료

2. ✅ **결과물① 품질검토 결과서 자동 생성**

   - **Excel (.xlsx) 형식**
   - ERROR/WARNING/OK 3단계 등급 명시
   - 항목별 발견 결함, 근거, 개선 방향 포함

3. ✅ **결과물② AI 개선 산출물 자동 생성** (단계별 1개)

   - **원본 포맷 유지** (HWP/HWPX 또는 Excel)
   - HWP → HWPX 업그레이드
   - 확인된 모든 ERROR 항목 수정
   - [개선] 태그로 변경 위치 명시

4. ✅ **결과물③ 시정조치관리대장 자동 생성** (매 검증마다)

   - **Excel (.xlsx) 형식**
   - 회차별 ERROR/WARNING 항목 정리
   - 조치 상태 추적 가능

5. ✅ **결과물④ 단계말 검토결과서 생성** (최종)

   - **HWPX (.hwpx) 형식**
   - QA 수기 확인 후 최종 승인
   - 고객사 제출용 공식 문서

6. ✅ **웹 UI 기본 완성** (정적 HTML/CSS + jQuery)

   - 파일 업로드 폼 동작 (HWP/HWPX/Excel)
   - 검토 진행도 표시
   - 결과물 4개 다운로드 링크
   - 회차별 현황 조회

7. ✅ **E2E 테스트 통과**

   - 11개 산출물 중 3개 이상으로 HWP/HWPX/Excel 혼합 검증
   - 4종 결과물이 합리적으로 생성됨을 시연
   - 단계별 프로세스 흐름 확인
   - 원본 포맷 유지 확인

### 💡 **추가 가치**

- AI 검증으로 "언제, 무엇이, 왜" 문제인지 명확히 기록
- 기존 수일 소요 → **수 분 단축** 입증
- 회차별 개선 추적으로 PM 역량 내재화 지원

## 12. 평가 기준별 부합성

### 12.1 혁신성

**평가:** ⭐⭐⭐⭐⭐

- 단순 검토 자동화 → **3종 결과물(검토서 + 개선원본 + 시정조치대장) 자동 생성**
- 관리·분석·설계 단계별 **체계적 검증 프로세스 자동화**
- PM이 AI를 '검토자'가 아닌 **'함께 산출물을 완성하는 파트너'로 활용**
- 국내 SI 업계에서 **전례 없는 접근법**

### 12.2 실현 가능성

**평가:** ⭐⭐⭐⭐⭐

- Claude API + Apache POI 등 **검증된 기술 조합으로 즉시 구현 가능**
- 사내 표준 체크리스트·양식이 이미 존재 → **품질 기준 진입장벽 낮음**
- **해커톤 기간(1개월) 내 프로토타입 시연 가능**
- Java/Spring Boot는 사내 표준 스택, **PostgreSQL은 오픈소스로 라이선스 비용 없이 JSONB 기반 체크리스트/검토결과 저장에 적합** → 해커톤 이후 SaaS화 확장 시에도 인프라 제약 최소

### 12.3 업무 적용성

**평가:** ⭐⭐⭐⭐⭐

- 현재 진행 중인 **모든 IT SI 프로젝트에 즉시 파일럿 적용 가능**
- PM·QA·관리자 모두 **실무에서 매일 사용하는 도구**
- **학습 비용 없음** (기존 파일 업로드 방식 활용)
- 품질검토 프로세스에 **즉시 통합 가능**

### 12.4 확장 가능성

**평가:** ⭐⭐⭐⭐

- **단기**: Word/PDF 형식, PMO 대시보드, 산출물 간 정합성
- **중기**: 테스트/이행 단계, 발주 유형별 특화
- **장기**: SaaS 서비스화, PM 컨설팅 상품, 산업별 패키지
